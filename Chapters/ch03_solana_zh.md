## 一、架构哲学：并行执行的激进押注

Solana 代表了一种与比特币和以太坊截然不同的区块链扩容思路。以太坊通过 Layer 2 Rollup 扩容（见第二章），保持基础层硬件要求较低以最大化去中心化。Solana 走了相反的路——**直接在 L1 层面追求极致吞吐量**，押注硬件性能的提升速度会快于区块链需求的增长。

### 核心创新：声明式账户访问 + 并行执行

```
以太坊模型（串行）：
  交易 A → 交易 B → 交易 C → ...
  像超市只有一条结账通道

Solana 模型（并行）：
  每笔交易必须预先声明要读写哪些账户
       │
       ▼
  运行时识别不冲突的交易 → 同时在多个 CPU 核心执行
       │
       ▼
  像超市同时开几十条结账通道

  更多 CPU 核心 = 更高交易吞吐量
```

### 账户模型

Solana 的状态组织将程序代码与用户数据严格分离：

| 类型 | 内容 | 特点 |
|------|------|------|
| **可执行账户** | 程序代码 | 实际上不可变 |
| **数据账户** | 用户级状态 | 由程序拥有和管理 |

程序之间通过**跨程序调用（CPI）** 实现可组合性——一个程序请求另一个程序执行操作，传入账户作为参数。

### 程序派生地址（PDA）

PDA 是 Solana 的独特创新——没有私钥的地址，只能由程序本身授权交易：

```
传统托管：人持有私钥 → 信任问题 + 单点故障
PDA 托管：程序控制资金 → 没有私钥可被窃取
```

## 二、交易、费用与用户体验

### 交易结构

每笔交易包含：消息（账户列表 + 指令 + 近期区块哈希）+ Ed25519 签名。

```
费用结构：

基础费：5,000 lamports/签名 ≈ $0.001
         │
         ├── 50% 永久燃烧
         └── 50% 给区块领导者

优先费：用户可选的计算单元费用
         │
         └── 100% 给当前领导者
```

### 局部费用市场 vs 全局费用市场

| 以太坊 | Solana |
|--------|--------|
| 全局费用市场：所有交易竞争相同区块空间 | **局部费用市场**：按账户级别定价拥堵 |
| 一个热门 App 可以让全网费用飙升 | 理想情况下，热门账户涨费不影响其他用户 |

> 实际上，2024-2025 年的极端垃圾交易事件中，拥堵仍会影响全局性能并导致大量交易被丢弃。局部费用市场的实现还在完善中。

### "丢弃"与"失败"的区别

| 类型 | 原因 | 链上记录 |
|------|------|---------|
| **丢弃（Dropped）** | 网络过载、优先费不足、区块哈希过期 | 无 |
| **失败（Failed）** | 程序逻辑错误、滑点超限等 | 有（被打包但回滚） |

### 用户体验优势

- **亚美分交易成本** → 用户可以频繁交易、小额投机、多次交互而无费用焦虑
- **近乎即时的交易处理** → 适合即时结算场景
- **原子可组合性** → 单笔交易中可组合多个协议交互，全部成功或全部失败
- **统一全局状态** → 无需在多链之间跳转

## 三、共识、调度与网络层

Solana 的快速确认来自多个系统的紧密集成：

### 历史证明（Proof of History）

PoH 是一个可验证的加密时钟——产生连续的哈希序列，让所有人在事件被添加到区块链之前就能就其相对顺序达成一致。

```
PoH 的作用：

传统共识：验证者需要大量通信来就顺序达成一致
PoH 共识：时间戳记录作为共享参考点 → 减少通信开销
```

### Tower BFT

基于 PoH 时间戳处理最终性。验证者对区块进行权益加权投票，PoH 时间戳帮助防止双重投票。当前最终性约 12.8 秒。

### 领导者调度与 Gulf Stream

```
时间单位：
  1 Slot ≈ 400 毫秒
  1 Epoch ≈ 2 天（领导者调度固定）

质押量决定被选为领导者的概率。

Gulf Stream（交易转发协议）：
  传统：交易广播到公共内存池 → 等待被选中
  Solana：交易直接发送给当前和即将到来的领导者
         → 跳过广播阶段，减少延迟
```

### Turbine（数据传播）

将区块拆分成小块（shreds），通过树状结构分发。包含冗余编码，即使部分 shreds 丢失也能重建完整区块。

### QUIC + 权益加权 QoS

底层传输使用 QUIC 协议，支持多路复用和更快的连接建立。质押量更大的验证者获得优先带宽——降低低质押者的垃圾交易能力。

### DoubleZero：专用光纤基础设施

连接验证者的私有光纤网络，绕过公共互联网的延迟波动。对 Alpenglow 的严格时间窗口要求至关重要。

### Alpenglow：下一代共识升级

```
Alpenglow 是 Solana 历史上最重大的协议级变更：

当前系统                              Alpenglow
─────────                            ──────────
Tower BFT 共识                    →   Votor（新投票方法）
PoH 作为时间基础                   →   固定 Slot 调度 + 本地计时器
~12.8 秒最终性                    →   ~100-150 毫秒最终性（模拟数据）
                                      20+20 弹性模型（容忍 40% 故障）
```

**Votor 的双路径最终性：**
- 第一轮 >=80% 质押支持 → 立即最终化
- 第一轮 60-80% → 进入第二轮，>=60% 则最终化

**Rotor** 优化区块数据传播——通过高质押可靠验证者直接路由消息。

**PoH 被弃用。** 这是 Solana 诞生以来最根本的架构变更。

目标主网上线时间：2026 年上半年（区块链升级历来容易延期）。

## 四、MEV 与区块构建

由于 Gulf Stream 将交易直接路由到领导者（而非公共内存池），Solana 的 MEV 格局与以太坊不同。

### Jito 生态

许多验证者运行 **Jito-Solana**（修改版客户端），支持 Bundle 拍卖。搜索者将交易打包成 Bundle，链下模拟后支付小费争取被包含。

**正在演进的区块构建层：**

| 项目 | 作用 |
|------|------|
| **BAM** | Jito 的交易管道重设计：TEE 隔离交易内容 → 验证者和构建者在排序前看不到原始交易 → 防止抢跑 |
| **Harmonic** | 开放的区块构建者聚合层：验证者实时接受多个构建者的区块提案 |
| **Raiku** | 确定性执行保证层：提供 AOT（预提交）和 JIT（即时）两种交易类型，面向高频交易和链上 CLOB |

## 五、经济模型、质押与治理

### 代币经济

```
SOL 通胀时间线：

初始通胀率：8%/年
每年递减：15%
终端通胀率：1.5%/年（约 2031 年达到）

通缩压力：50% 基础费永久燃烧
极端活跃期 → 燃烧可能超过通胀（但当前不常态）
```

### 质押机制

| 维度 | Solana | 以太坊（对比） |
|------|--------|-------------|
| 最低质押 | 无最低限制（原生委托任意金额） | 32 ETH |
| 委托模式 | 原生委托，不转移托管权 | 需流动性质押协议（Lido 等） |
| 激活/停用 | Epoch 边界（约 2-3 天） | 退出队列（可能数天到数周） |
| 验证者佣金 | 通常 0-10% | 由协议定义 |
| 质押收益 | ~7% APY | ~3-4% APY |

### 验证者经济

运营一个 Solana 验证者成本高昂：

```
月度成本构成：
  高端硬件折旧 + 数 TB 带宽 + 企业网络 + 数据中心 ≈ ~$1,000
  投票交易费 ≈ ~$4,000
  ──────────────────
  总计 ≈ ~$5,000/月
```

收入来源：通胀奖励（按质押权重分配）+ 交易费（基础费 50% + 优先费）+ MEV 小费（运行 Jito 时）。

### 中心化压力

```
验证者数量趋势：
  峰值 ~2,000 → 2026年1月 ~800

超级少数（Superminority）：
  仅 19-22 个大型验证者控制 ~1/3 质押
  → 如果协调行动，可停止网络出块
```

**关键问题：** Solana 主网当前**不执行罚没**。验证者不会因为投票冲突或长期离线而自动损失质押 SOL。网络更多依赖声誉和经济机会成本来维持诚实行为。

### 治理

无绑定性链上投票。通过 **SIMD**（Solana 改进文档）流程进行协议变更。Solana 基金会、Anza（Agave 客户端维护者）、Jump Crypto、Jito Labs 等拥有较大非正式影响力。

协议变更通过**特性门控（Feature Gates）** 实施——默认禁用的协议更改，在超级多数质押加权验证者升级后通过链上指令激活。

## 六、开发者工具与标准

### 执行环境（SVM）

```
以太坊 EVM（基于栈）     vs     Solana SVM（基于寄存器）
  像叠盘子，只能操作顶部           像 CPU，值存在编号寄存器中
  适合通用性                       更适合高性能并行执行
```

程序主要用 Rust 编写，编译为可移植指令格式。运行在严格沙箱中——无文件系统访问、无网络访问、不可逃逸。

**Anchor 框架**是事实标准开发工具包——自动生成 IDL、验证账户、提供常见操作的标准化模式。

### SPL 代币：共享基础设施 vs 独立部署

```
以太坊模式：
  每个代币 = 独立的智能合约
  → 数千种实现，各有差异

Solana 模式：
  所有代币共享同一个 SPL Token 程序
  创建新代币 = 创建一个 Mint 账户（非部署新代码）
  → 核心程序优化时，所有代币立即受益
```

**关联代币账户（ATA）** 为每个钱包-代币对自动派生标准地址，消除了用户手动创建代币账户的整类错误。

**Token-2022** 在保持向后兼容的同时增加可编程功能：转账钩子（自动版税/合规检查）、机密转账（加密证明隐私）等。

### 压缩 NFT

```
传统 NFT：每个需独立链上账户
  100 万个 NFT ≈ $250,000 账户租金

压缩 NFT（状态压缩）：
  链上：单个并发默克尔树根
  链下：详细元数据
  验证：默克尔证明
  100 万个 NFT ≈ < $100
```

## 七、性能与其代价

### 数据增长挑战

| 指标 | 数值 |
|------|------|
| 完整存档账本 | ~350 TB |
| 年增长 | ~90 TB |
| 存档节点月成本 | ~$40,000 |

但普通验证者和 RPC 节点通过修剪历史数据大幅降低存储需求。长期历史数据交给专业存档基础设施。

### 客户端多样性与 Firedancer

Solana 历史上依赖单一代码库（Agave 及其分叉 Jito-Solana）。**Firedancer** 是 Jump Crypto 用 C 语言从零开始重写的独立验证者实现——直接解决这一单点故障。

```
客户端演进路线：

Agave（Rust）─── 当前生产环境主力
     │
Frankendancer ── 过渡版：Firedancer 网络层 + Agave 运行时
     │                  2024.9 上线部分验证者
     │
Firedancer（C）── 目标：完全独立实现
                  截至 2026 初仍需 Agave 组件
```

## 八、适用场景与局限

### Solana 擅长的领域

- **Meme 币交易**：亚美分费用 + 即时确认 = 快速投机的理想环境（Pump.fun、Jupiter）
- **移动优先体验**：Phantom、Moonshot 等原生移动应用
- **链上 CLOB（中央限价订单簿）**：传统 L1 因慢出块和高费用无法支持频繁的挂单/撤单
- **主动型 AMM**：持续更新价格追踪外部市场的链上做市商
- **代币化证券**：xStocks 等将传统股票带到链上

### 局限性

| 维度 | 说明 |
|------|------|
| **去中心化** | 高硬件要求限制了验证者准入，质押集中度高 |
| **开发语言** | Rust 生态对习惯 Solidity 的开发者有学习曲线 |
| **流动性** | 网络效应和先发优势使某些深流动性仍集中在以太坊 |
| **正常运行时间** | 历史上有数次停机事件，机构用户需评估风险容忍度 |
| **极端性能需求** | Hyperliquid 等仍选择专用链——通用 L1 有其极限 |

### 2026 升级愿景

```
Alpenglow（亚秒最终性）
  + Firedancer（百万级 TPS 潜力）
  + DoubleZero（专用光纤）
  + BAM + Harmonic + Raiku（MEV 公平性 + 确定性执行）
  ──────────────────────
  = 目标：去中心化的纳斯达克
```

能否实现取决于升级是否按时交付——区块链行业的重大协议升级历来容易延期。

## 章节要点速览

1. **Solana 通过声明式账户访问实现并行执行——根本性地不同于以太坊的串行模型**
2. **亚美分费用和亚秒确认创造了截然不同的用户行为——尤其利好高频交易和小额投机**
3. **局部费用市场按账户级别定价拥堵，但实际实现仍不完美**
4. **PoH + Tower BFT + Gulf Stream + Turbine 构成当前的集成共识和网络栈**
5. **Alpenglow 将弃用 PoH，目标 100-150 毫秒最终性——Solana 历史上最根本的协议变更**
6. **验证者运营成本高（~$5,000/月），主网无罚没，超级少数仅 19-22 个验证者**
7. **SPL Token 的"共享程序"模式与以太坊"每个代币一个合约"形成鲜明对比**
8. **Firedancer 独立客户端实现正在解决 Solana 最大的架构弱点——客户端单一性**
9. **Solana 的终极愿景是成为"去中心化纳斯达克"，但高硬件要求与去中心化之间的张力始终存在**
